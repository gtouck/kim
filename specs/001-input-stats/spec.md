# Feature Specification: 用户输入统计工具

**Feature Branch**: `001-input-stats`
**Created**: 2026-03-06
**Status**: Draft
**Input**: User description: "我想要构建一个用户输入统计工具，该工具可以统计我每天键盘敲击次数、鼠标点击次数；还能够统计我每天的打字数量（不考虑粘贴的文字），该统计依赖我的输入法，比如我是中文用户，则输入一个中文算一个字，输入一个英文字母也算一个字；还能够统计我每日ctrl-c，ctrl-v的次数；该软件在点击运行后最好能够一直在后台运行，可以拥有简单的命令行界面进行统计信息的显示。"

## Clarifications

### Session 2026-03-06

- Q: 「打字字符数」的判定边界是什么？ → A: 仅统计「产生可见文本输出」的字符——通过 IME 提交的字符 + 直接键入的英文/数字/标点符号，排除功能键（F1-F12、Esc、方向键等）和快捷键组合
- Q: 隐私保护范围如何处理密码输入？ → A: 当焦点在密码输入字段时，该输入不计入打字字符数，但仍计入键盘敲击次数；其余场景完整统计
- Q: 工具是否需要支持开机自启动？ → A: 支持开机自启，但默认关闭；用户可通过 CLI 命令开启或关闭自启功能
- Q: 统计数据的持久化写入频率是多少？ → A: 每 30 秒批量写入一次，最多丢失 30 秒内的计数
- Q: Backspace 删除已提交字符时，打字字符数如何计算？ → A: 总提交量，不扣除删除——Backspace 不影响打字字符数，已提交字符永久计入
- Q: 按应用统计输入量时，「应用」的识别粒度是什么？ → A: 按可执行文件名（进程名）区分，如 Code.exe、chrome.exe
- Q: 编程语言如何检测？编程时间如何定义？ → A: 从活动窗口标题解析文件扩展名推断语言（.py→Python、.ts→TypeScript 等）；当对应编辑器窗口处于前台焦点时计为该语言的编程时间（包括不按键的浏览/思考时段）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 后台静默监控输入 (Priority: P1)

用户启动工具后，工具在系统后台静默运行，持续监控并记录所有键盘敲击、鼠标点击及输入法提交的字符，不打扰当前工作流。即使在后台运行，工具进程也在系统重启后保留历史统计数据。

**Why this priority**: 这是整个工具的核心能力。没有后台持续监控，所有其他统计功能无从实现。它也是使用门槛最低的功能——用户启动一次即可受益。

**Independent Test**: 启动工具后，在任意文本编辑器中输入若干文字、点击鼠标，再通过 CLI 查询统计，验证数字正确增加。

**Acceptance Scenarios**:

1. **Given** 工具尚未运行，**When** 用户执行启动命令，**Then** 工具进程在后台启动，不显示任何弹窗，命令提示符立即返回可用状态
2. **Given** 工具正在后台运行，**When** 用户在任意应用中按下键盘按键，**Then** 该次按键在 1 秒内被记录到当日键盘敲击计数中
3. **Given** 工具正在后台运行，**When** 用户点击鼠标（左键、右键、中键），**Then** 该次点击在 1 秒内被记录到当日鼠标点击计数中
4. **Given** 工具已记录当日部分数据，**When** 当日结束（本地时间 00:00），**Then** 当日统计数据被归档，次日统计从零开始
5. **Given** 工具已记录数据，**When** 操作系统重启后用户再次启动工具，**Then** 历史统计数据仍然可查，不丢失

---

### User Story 2 - 准确统计实际打字字符数 (Priority: P2)

用户（中文输入法用户）希望统计自己每天实际输入的字符数量。输入一个中文字算一个字，输入一个英文字母也算一个字，粘贴的内容不计入打字数量。

**Why this priority**: 打字数量是衡量工作量的核心指标，且涉及输入法集成，是工具区别于普通按键计数器的关键特性。

**Independent Test**: 使用中文输入法连续输入 3 个中文字，再输入 5 个英文字母，通过 CLI 验证打字数显示为 8。

**Acceptance Scenarios**:

1. **Given** 工具后台运行，**When** 用户通过中文输入法提交一个中文字符（如"你"），**Then** 打字数计数增加 1，同时键盘敲击数增加若干次（对应输入拼音的按键次数）
2. **Given** 工具后台运行，**When** 用户直接键入产生可见文本的按键（英文字母、数字、标点符号，如"a"、"1"、"."），**Then** 打字数计数增加 1，键盘敲击数也增加 1
3. **Given** 工具后台运行，**When** 用户按下功能键（如 F5、方向键、Esc、Home）或快捷键组合（非 Ctrl+C/V），**Then** 键盘敲击数增加 1，但打字字符数不增加
4. **Given** 工具后台运行，**When** 用户执行 Ctrl+V 粘贴文本，**Then** 粘贴的字符数量不计入打字数，打字数保持不变
5. **Given** 工具后台运行，**When** 用户在密码输入字段（如登录框、密码管理器）中输入字符，**Then** 键盘敲击数正常增加，但打字字符数不增加（隐私保护）
6. **Given** 工具后台运行，**When** 用户输入若干字符后按 Backspace 删除，**Then** 打字字符数不因 Backspace 减少，已计入的字符数永久保留

---

### User Story 3 - 统计剪贴板快捷键操作次数 (Priority: P2)

用户希望统计自己每天执行了多少次复制（Ctrl+C）和粘贴（Ctrl+V）操作，以了解粘贴行为频率。

**Why this priority**: 这是用户明确要求的统计维度，且实现成本低，属于监控逻辑的自然延伸。

**Independent Test**: 连续执行 3 次 Ctrl+C 和 2 次 Ctrl+V，通过 CLI 验证复制次数为 3，粘贴次数为 2。

**Acceptance Scenarios**:

1. **Given** 工具后台运行，**When** 用户按下 Ctrl+C，**Then** 当日复制次数增加 1
2. **Given** 工具后台运行，**When** 用户按下 Ctrl+V，**Then** 当日粘贴次数增加 1，且对应文字不计入打字数
3. **Given** 工具后台运行，**When** 用户使用右键菜单"复制"或"粘贴"（非快捷键），**Then** 此类操作不计入 Ctrl+C/V 计数（本工具仅统计快捷键操作）

---

### User Story 4 - CLI 查看统计信息 (Priority: P1)

用户希望通过命令行界面查看今日及历史各日的输入统计数据，包含键盘敲击数、鼠标点击数、打字数、复制次数和粘贴次数。

**Why this priority**: 统计数据的展示是用户获取价值的最终出口，没有展示能力则所有监控均无意义。

**Independent Test**: 仅通过 CLI 命令即可看到格式化的当日统计数据，无需安装任何 GUI 工具。

**Acceptance Scenarios**:

1. **Given** 工具后台运行，**When** 用户执行查询今日统计的命令，**Then** 在 1 秒内输出包含「键盘敲击数、鼠标点击数、打字字符数、复制次数、粘贴次数」的格式化表格
2. **Given** 工具已积累多日数据，**When** 用户查询历史某日（如昨日）统计，**Then** 正确显示该日数据
3. **Given** 工具尚未启动，**When** 用户尝试通过 CLI 查询统计，**Then** 提示工具未运行及启动方式，不输出错误堆栈
4. **Given** 工具后台运行，**When** 用户执行停止命令，**Then** 后台监控进程安全退出，数据落盘不丢失
5. **Given** 工具已安装，**When** 用户通过 CLI 开启开机自启，**Then** 系统重启后工具自动在后台启动，无需手动操作
6. **Given** 工具后台运行，**When** 用户通过 CLI 关闭开机自启，**Then** 下次系统重启时工具不再自动启动

---

### User Story 5 - 按应用统计输入量 (Priority: P3)

用户希望了解自己每天在哪些软件中打了多少字、敲了多少键，以掌握各应用的使用强度（如"今天在 VS Code 中输入了 3000 字"）。

**Why this priority**: 应用维度的分解让整体统计数据更有意义，是核心统计功能的自然扩展。P3 是因为它依赖核心监控能力（P1）先完成。

**Independent Test**: 在 VS Code 和记事本中各输入若干文字，通过 CLI 查询按应用统计，验证两个应用分别显示正确的输入字符数和敲击数。

**Acceptance Scenarios**:

1. **Given** 工具后台运行，**When** 用户在某应用（如 Code.exe）中输入字符，**Then** 该次输入同时计入全局统计和该应用的独立统计中
2. **Given** 工具已积累多个应用的数据，**When** 用户执行按应用汇总的 CLI 命令，**Then** 输出各应用（进程名）当日的键盘敲击数、打字字符数排行
3. **Given** 工具后台运行，**When** 用户切换到另一应用继续输入，**Then** 新输入归属于新应用，不影响上一应用的统计
4. **Given** 某应用当日无任何输入，**When** 用户查询按应用统计，**Then** 该应用不出现在结果列表中（只展示有数据的应用）

---

### User Story 6 - 统计编程语言输入量与专注时间 (Priority: P3)

用户希望了解自己每天在不同编程语言上投入了多少文字输入量和专注时间（如"今天写 Python 2 小时，打了 800 字"），类似 WakaTime 但完全本地运行。

**Why this priority**: 对开发者用户而言这是高价值功能，但依赖应用统计（User Story 5）的基础能力，故同为 P3。

**Independent Test**: 在编辑器中打开 `test.py` 并输入若干字符，然后切换到 `index.ts` 继续输入，通过 CLI 验证 Python 和 TypeScript 各自显示正确的字符数和专注时长。

**Acceptance Scenarios**:

1. **Given** 工具后台运行，**When** 用户在编辑器中编辑扩展名为 `.py` 的文件，**Then** 该时段的输入字符数计入 Python 语言统计，专注时间从窗口获得焦点时开始计算
2. **Given** 工具后台运行，**When** 活动窗口标题中含有可识别的文件扩展名，**Then** 工具根据扩展名映射推断编程语言并更新对应语言的专注时间
3. **Given** 工具后台运行，**When** 活动窗口标题不含文件扩展名或扩展名不在已知映射列表中，**Then** 该时段归类为「其他/未知」，不影响其他语言统计
4. **Given** 工具已积累多日数据，**When** 用户通过 CLI 查询编程语言统计，**Then** 输出各语言当日的输入字符数和专注时间（小时:分钟），按专注时间降序排列

---

### Edge Cases

- 系统锁屏或休眠期间，监控进程暂停记录，恢复后继续；锁屏期间无法输入，故不影响统计
- 键盘按键持续按住产生的连击（key repeat），每次系统触发的按键事件均计为一次敲击
- 用户同时使用外接键盘和内置键盘时，两者输入均被统计
- 若后台进程意外崩溃，工具应保留崩溃前已持久化的数据，用户重启工具后可继续统计；由于数据每 30 秒写入一次，最多丢失 30 秒内的计数
- 当天结束前后（00:00 附近）产生的输入事件，应归属于对应的正确日期
- 密码字段检测依赖操作系统的无障碍辅助功能（Accessibility API）；若无法检测（某些非标准控件），则该字段内容默认仍计入键盘敲击数，打字字符数不增加（保守保护策略）
- 同一进程名的多个窗口（如同时打开多个 Chrome 窗口）的输入量合并统计到同一进程名下
- 当窗口标题快速切换（如 IDE 文件标签频繁切换）时，专注时间应归属于持续聚焦时长较长的文件，避免毫秒级切换产生大量噪声记录（建议最短计入阈值：连续聚焦超过 5 秒才计入该语言的专注时间）
- 若活动窗口标题格式为「文件名 - 应用名」，应正确解析文件名部分；若格式无法解析，归为「未知」类- Q: 实现语言是什么？ → A: Rust — 运行时轻量，通过 WinAPI 绑定实现全局钩子，IME 支持需自行实现
- Q: 最低 Windows 版本要求是什么？ → A: Windows 10 及以上
- Q: 统计数据本地持久化使用什么存储格式？ → A: SQLite 单文件数据库（通过 `rusqlite` crate 操作）
- Q: 后台进程是否需要系统托盘图标？ → A: 无托盘，纯 CLI 控制——后台进程静默运行，所有交互通过 CLI 命令完成（start/stop/status）
- Q: CLI 二进制名称与进程间通信方式是什么？ → A: 单二进制 `kim`；daemon 启动后写入 PID 文件；CLI 查询直接读 SQLite；stop 命令通过 PID 终止进程
## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须能够在 Windows 操作系统上以后台进程方式运行，启动后不占用终端窗口
- **FR-002**: 系统在后台运行时，必须监控并记录全局键盘按键事件（包含所有按键类型，不限于字母数字键）
- **FR-003**: 系统在后台运行时，必须监控并记录全局鼠标按键点击事件（左键、右键、中键均计入）
- **FR-004**: 系统必须统计「产生可见文本输出」的字符数量作为打字字符数，具体包括：(1) 用户通过输入法（IME）最终提交的字符（每个提交字符计为 1），(2) 用户直接键入的英文字母、数字及标点符号（每次按键计为 1）；功能键（F1-F12、Esc、方向键、Home/End 等）和快捷键组合不计入打字字符数
- **FR-004a**: 当系统检测到当前焦点位于密码输入字段时，该字段内的按键不计入打字字符数，但仍计入键盘敲击总次数（隐私保护）
- **FR-005**: 通过 Ctrl+V 粘贴的文本内容，必须不被计入打字字符数
- **FR-006**: 系统必须记录每日 Ctrl+C 快捷键的使用次数
- **FR-007**: 系统必须记录每日 Ctrl+V 快捷键的使用次数
- **FR-008**: 系统必须在本地时间 00:00 后将当日统计数据归档，并将次日计数重置
- **FR-009**: 所有统计数据必须持久化存储于本地，每 30 秒至少写入一次，工具重启或系统重启后数据不丢失；正常退出时必须立即执行最终写入
- **FR-010**: 系统必须保留最近 30 天的历史统计数据
- **FR-011**: 系统必须提供 CLI 命令查看今日统计数据
- **FR-012**: 系统必须提供 CLI 命令查看历史指定日期的统计数据
- **FR-013**: 系统必须提供 CLI 命令安全停止后台监控进程（数据落盘后退出）；`kim stop` 通过读取 PID 文件向 daemon 发送终止信号
- **FR-014**: 系统必须提供 CLI 命令查询后台进程运行状态（运行中 / 已停止）；`kim status` 通过检查 PID 文件及进程是否存活来判断
- **FR-015**: 系统必须提供 CLI 命令开启开机自启动（默认关闭）
- **FR-016**: 系统必须提供 CLI 命令关闭开机自启动
- **FR-017**: 系统在后台运行时，必须持续追踪当前前台活动窗口的进程名（可执行文件名）
- **FR-018**: 所有输入事件（键盘敲击、打字字符、Ctrl+C/V）必须同时按进程名维度分类记录，不仅记录全局总计
- **FR-019**: 系统必须提供 CLI 命令查看当日按应用（进程名）分组的输入统计，包含各应用的键盘敲击数和打字字符数
- **FR-020**: 系统必须从活动窗口标题中解析文件扩展名，并根据内置扩展名→语言映射表推断当前编程语言（至少支持：.py、.js、.ts、.java、.go、.rs、.c、.cpp、.cs、.rb、.php、.swift、.kt、.html、.css、.sql、.sh）
- **FR-021**: 当编辑器窗口连续持有前台焦点超过 5 秒时，该时段计入对应编程语言的专注时间；切换窗口时立即停止计时
- **FR-022**: 系统必须提供 CLI 命令查看当日编程语言统计，包含各语言的打字字符数和专注时间，按专注时间降序排列
- **FR-023**: 无法识别语言的编辑器活动（窗口标题不含已知扩展名）归类为「其他」，不影响有语言标签的统计

### Key Entities

- **DailyStats（日统计记录）**: 代表某一日期的完整输入统计，包含：日期、键盘敲击总次数、鼠标点击总次数、打字字符数、Ctrl+C 次数、Ctrl+V 次数
- **AppStats（应用统计记录）**: 某一日期、某一应用（进程名）维度的输入统计，包含：日期、进程名、键盘敲击次数、打字字符数、Ctrl+C 次数、Ctrl+V 次数
- **LanguageStats（语言统计记录）**: 某一日期、某一编程语言维度的统计，包含：日期、语言名称、打字字符数、专注时长（秒）
- **InputEvent（输入事件）**: 系统捕获到的原始输入事件，用于实时累加到当日统计；最终持久化时合并到 DailyStats 和 AppStats
- **ProcessState（进程状态）**: 后台监控进程的运行状态信息，用于 CLI 判断进程是否存活

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 工具启动后，键盘和鼠标事件的统计延迟不超过 1 秒（事件发生到数据更新完成）
- **SC-002**: 使用中文输入法输入 10 个中文字后，打字字符数准确显示为 10，误差为 0
- **SC-003**: 执行 10 次 Ctrl+V 粘贴操作后，打字字符数不因此增加，复制/粘贴计数准确无误
- **SC-004**: CLI 查询命令响应时间不超过 1 秒
- **SC-005**: 工具连续运行 8 小时后，内存占用增长不超过初始值的 50%（无持续性内存泄漏）
- **SC-007**: 数据持久化写入操作不阻塞输入事件处理，写入期间 CPU 占用峰值不超过 5%
- **SC-006**: 工具运行期间，用户在任意应用中感知不到任何输入延迟或操作响应变慢
- **SC-008**: 按应用统计中，同一应用的输入量累计误差为 0（不因窗口切换产生计数丢失）
- **SC-009**: 编程语言专注时间的计时精度误差不超过 5 秒/小时（即长时间工作后累积误差可控）

## Assumptions

- **目标平台**: 仅支持 Windows 操作系统（基于用户当前使用环境）
- **连击计数**: 键盘按键持续按住触发的 key repeat 事件，每次均计为一次敲击
- **删除字符计数**: 打字字符数按「总提交量」计算，Backspace 不影响打字字符数——已提交字符永久计入，无论后续是否被删除
- **历史数据保留**: 本地保留最近 30 天的历史统计，更早的数据自动清理
- **非快捷键剪贴板操作**: 仅统计 Ctrl+C / Ctrl+V 快捷键，不额外监听右键菜单等方式触发的剪贴板操作
- **管理员权限**: 系统级全局输入监控能力在某些系统配置下可能需要特定权限，工具文档应说明运行所需权限
- **开机自启**: 默认不注册开机启动，用户可通过 CLI 显式开启；开机自启通过操作系统标准机制实现（如 Windows 注册表启动项或任务计划程序）
- **多用户环境**: 工具仅统计当前登录 Windows 用户的输入，不支持多用户同时统计
- **实现语言**: 使用 Rust 实现，通过 `windows-rs` 或 `winapi` crate 绑定 WinAPI（`WH_KEYBOARD_LL`、`WH_MOUSE_LL` 钩子），IME 字符提交检测通过 `WM_IME_COMPOSITION` 消息自行实现；CLI 工具同样用 Rust 编写
- **最低 Windows 版本**: Windows 10 及以上；不支持 Windows 7/8/8.1
- **数据存储格式**: 使用 SQLite 单文件数据库（`rusqlite` crate）；数据库文件存放于用户数据目录（如 `%APPDATA%\key-input-monitor\stats.db`）；包含 `daily_stats`、`app_stats`、`language_stats` 三张表，对应规格中的 Key Entities
- **后台进程可见性**: 无系统托盘图标，后台进程静默运行；所有交互（启动、停止、状态查询、数据展示）均通过 CLI 命令完成，不引入 GUI 依赖
- **CLI 架构**: 单一可执行文件 `kim`（如 `kim.exe`）；`kim start` 将自身作为后台进程启动并写入 PID 文件（`%APPDATA%\key-input-monitor\kim.pid`）；`kim stop` 读取 PID 文件并发送终止信号；CLI 查询命令（`today`、`history`、`apps`、`langs`）直接读取 SQLite 数据库，无需与 daemon 通信
- **应用识别粒度**: 以可执行文件名（进程名）为应用唯一标识，同一进程名的多窗口合并统计
- **语言检测方式**: 从活动窗口标题解析文件扩展名推断编程语言；连续聚焦超过 5 秒才开始计入该语言的专注时间，避免快速切换产生噪声
- **语言映射维护**: 扩展名→语言映射表内置于工具中，用户无需手动配置；未来可通过配置文件扩展
